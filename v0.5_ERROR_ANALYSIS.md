## 🔴 HIGH SEVERITY (Will Cause Crashes/Data Corruption)

### Issue #1: 3-tuple unpacking crash in generate_tab.py
**File:** `app/datahub/generate_tab.py:441`  
**Type:** Logic/Type Error  
**Severity:** CRITICAL

**Problem:**
```python
# Line 441: unpacks as 2-tuple
for _, s in results:  # results[i] is (P, Q, status) — 3 elements!
    # ValueError: too many values to unpack
```

`build_kernels()` returns `list[tuple[int, int, str]]` (P, Q, status), but the code unpacks it as 2-tuple. Every kernel generation task crashes with `ValueError: too many values to unpack`.

**Impact:** Kernel generation feature is completely broken.

**Fix:**
```python
for _, _, s in results:  # unpack 3-tuple
```

### Issue #2: WeylWorker stale-signal guard missing
**File:** `app/pipeline/index_card.py:357–377`  
**Type:** Thread Safety / Race Condition  
**Severity:** CRITICAL

**Problem:**
```python
def _on_weyl_finished(self, payload: dict) -> None:
    # ❌ NO generation counter check!
    self._session.weyl_result  = ab  # stale data written to new session
    self._session.weyl_checked = True
```

When a user loads a new manifold while Weyl check is in flight:
1. `lock()` disconnects the worker's signal
2. But a signal queued in Qt's event loop before disconnect completes will still execute
3. Stale vectors are written into the **new** session object
4. Session is corrupted with wrong data

**Fix:** Add generation counter check (like NCWorker and FillWorker):
```python
def _on_weyl_finished(self, payload: dict, gen: int) -> None:
    if gen != self._session_gen:
        return  # stale
```

### Issue #6: Module-level caches not thread-safe
**Files:**  
- `core/dehn_filling.py:63` (`_summation_term_cache`)
- `core/refined_dehn_filling.py:128` (`_tet_arr_cache`), `188` (`_iref_cache`)
- `core/kernel_cache.py:529` (`_kernel_mem_cache`)

**Type:** Thread Safety  

**Problem:**
```python
_kernel_mem_cache: dict = {}  # plain dict, no lock

# Multiple threads (DataHub batch workers) access simultaneously:
# Thread 1: if k not in cache: cache[k] = ...  # read, then write
# Thread 2: if k not in cache: cache[k] = ...  # read, then write
```

CPython's GIL protects individual dict operations, but compound patterns like `cache[k] = cache.get(k, []) + new_item` are not atomic. Concurrent DataHub workers corrupting these caches.

**Fix:** Add `threading.Lock()` and protect reads:
```python
_kernel_mem_cache: dict = {}
_kernel_mem_cache_lock = threading.Lock()

with _kernel_mem_cache_lock:
    if k not in _kernel_mem_cache:
        _kernel_mem_cache[k] = value
```

## 🟡 LOW SEVERITY (Edge Cases, Logic, Dead Code)

### Issue #7: Summary shows "full refined" for manifolds with no hard edges
**File:** `app/pipeline/index_card.py:407`

**Problem:**
```python
mode = "full refined" if all(self._active_edges()) else ...
```

When `num_hard == 0`, `_active_edges()` returns `[]` and `all([])` is `True` in Python. Shows "full refined" instead of "3D index".

**Fix:**
```python
mode = "full refined" if (num_hard > 0 and all(self._active_edges())) else ...
```

### Issue #9: Even nc_P bypass discards basis-change computation
**File:** `services/filling_service.py:183–186`

### Issue #14: closeEvent lacks interrupt support for workers
**File:** `app/window.py:256–272`

**Problem:**
```python
for worker in self._workers:
    worker.wait(2000)  # blocks up to 2 sec per worker
    if hasattr(worker, 'cancel'):
        worker.cancel()  # only LoadWorker has this
```

IndexWorker, WeylWorker, NCSearchWorker, FillWorker have no `cancel()` method. Long computations block shutdown for 2 seconds with no interrupt option.

**Fix:** Add `cancel()` method to all worker classes or use `worker.terminate()`.
